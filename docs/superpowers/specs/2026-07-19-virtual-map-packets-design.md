# Virtual Map Packets Design

## Goal

Store and display camera photos without creating Bukkit `MapView` objects or
Paper `MapItemSavedData` entries, while ensuring every plugin-owned virtual map
ID can never conflict with an ID generated by vanilla Minecraft.

## Identity

`VirtualMapIdRepository` persists a plugin-owned monotonically increasing ID
range in `photos.db`.  On first startup it reads the server's vanilla map-ID
high-water mark through a small Paper/NMS adapter, starts above that value plus
a fixed safety offset, and persists the chosen lower bound before issuing an
ID.  The adapter also advances vanilla's map-ID counter past the plugin-owned
upper bound before any virtual ID is issued.  Consequently later vanilla map
creation always allocates an ID greater than every previously allocated virtual
ID.

The repository never reuses an ID.  Existing `PhotoRecord.mapIds` values remain
the stable virtual IDs for each photo tile; the photo-bag preview is assigned
one additional virtual ID.

## Rendering

Photo map items contain their virtual map ID in the normal `minecraft:map_id`
component plus the existing TobysCamera identity tags.  No code calls
`Bukkit.createMap`, `Bukkit.getMap`, or attaches a Bukkit `MapRenderer` for a
camera photo.

When a source becomes active, `VirtualMapUpdateService` asynchronously obtains
the immutable 128x128 palette tile.  It sends a complete vanilla
`ClientboundMapItemDataPacket` only to players whose active source references
that tile.  The packet uses the source item's virtual ID, fixed scale and lock
flags, no decorations, and a full 128x128 patch.  The service deduplicates a
tile per player and revision, so unchanged still photos are not resent.

## Activation and lifetime

The existing hand and viewer-chunk tracking continues to determine activation.
Hand sources identify their player directly.  Frame sources include the viewer
player ID already present in their source key.  On final source removal, the
service drops its pixel reference; no native map data exists to evict.

## Safety and compatibility

The virtual-ID range is disjoint from vanilla IDs in both directions: the
initial range begins above the current vanilla counter and the vanilla counter
is advanced beyond every allocated virtual ID.  Allocation is transactional in
SQLite, so a restart cannot repeat an ID.

Vanilla clients receive standard map-update packets and require no mod.
Camera maps are marked with existing custom-data identity and their cartography
copy/scale behaviour remains controlled by the plugin.  Plugins that require
`Bukkit.getMap` for a camera map are not supported because the virtual ID has
no Bukkit `MapView`.

## Verification

Unit tests prove persisted allocation is monotonic, survives repository
reopening, and starts strictly above a supplied vanilla high-water mark.  Tests
also prove a virtual allocation advances the vanilla allocator floor and that
photo records receive virtual IDs without invoking map creation.  Packet tests
verify a complete 128x128 patch uses the requested virtual ID and a duplicate
revision is not sent twice to the same player.
